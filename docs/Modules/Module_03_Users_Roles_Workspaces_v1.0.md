# Module_03_Users_Roles_Workspaces_v1.0

Дата фиксации: 20.06.2026

Статус: архитектурный фундамент доступа принят.

Основной документ границ доступа: `docs/Access/ATOM_Role_Matrix_v1.3.md`.

## Назначение

Module_03 отвечает за:

- пользователей;
- роли;
- права доступа;
- рабочие пространства;
- персональные дашборды;
- аудит действий;
- безопасность платформы.

Модуль является фундаментом всей системы доступа.

## Главный принцип

Не существует единого интерфейса для всех.

Каждый пользователь должен видеть только то, что относится к его роли, правам и рабочему пространству.

Номер модуля является человекочитаемой меткой. Системная логика должна
переходить на `moduleCode`, `moduleId` и `canonicalModuleCode` после спринтов
PlatformModuleRegistry, PlatformRegionRegistry и auth-ready режима.

Первый MVP работает по пилотному региону Краснодарский край, но регион нельзя
хардкодить. Регион должен приходить из `PlatformRegionRegistry` или совместимого
adapter-слоя.

Схема:

```text
User
↓
Role
↓
Permission
↓
Workspace
↓
DashboardProfile
↓
Modules
↓
Functions
```

## Пользователь

Для пользователя хранить:

- id;
- email;
- телефон;
- имя;
- фамилию;
- статус;

Поле `User.role` остается для обратной совместимости текущих API.

## Роли

Роли должны быть настраиваемыми.

Базовый справочник:

- `ADMIN`;
- `MANAGER`;
- `ESTIMATOR`;
- `ENGINEER`;
- `SUPPLIER`;
- `CONTRACTOR`;
- `CUSTOMER`;
- `ANALYST`;
- `MODERATOR`;
- `VIEWER`.

Администратор сможет создавать новые роли без изменения кода.

## Права

Права назначаются отдельно от ролей.

Целевая структура Permission:

```text
moduleCode
actionCode
accessLevel
accessScope
conditions
isActive
```

`moduleCode` ссылается на `PlatformModuleRegistry.moduleCode`, а не на
визуальный номер модуля.

Примеры:

- `VIEW_MATERIALS`;
- `EDIT_MATERIALS`;
- `DELETE_MATERIALS`;
- `VIEW_DOCUMENTS`;
- `EDIT_DOCUMENTS`;
- `VIEW_PRICES`;
- `EDIT_PRICES`;
- `VIEW_USERS`;
- `MANAGE_USERS`;
- `VIEW_AUDIT`;
- `VIEW_KNOWLEDGE`;
- `EDIT_KNOWLEDGE`.

## Доступ к модулям

`ModuleAccess` задает уровень доступа:

- `NO_ACCESS`;
- `VIEW`;
- `CREATE`;
- `EDIT`;
- `APPROVE`;
- `ADMIN`.

Scope доступа хранится отдельно от уровня действия. Значения вроде `VIEW_OWN`,
`ADMIN_OWN`, `LIMITED_VIEW` и `VIEW_RELEVANT` не являются `AccessLevel`.

Целевая модель проверки:

```text
AccessLevel + AccessScope
```

Примеры scope:

- `NONE`;
- `GLOBAL`;
- `OWN`;
- `RELEVANT`;
- `LIMITED`.

Доступ может быть назначен роли, пользователю или рабочему пространству.

## Доступ к функциям

`FunctionAccess` задает права внутри модуля.

Пример:

пользователь может смотреть материалы, но не может редактировать или удалять.

## Переходные спринты Module_03

### Sprint 1.1. PlatformRegionRegistry и пилотный регион

Нужно подготовить справочник регионов платформы.

Первый MVP использует пилотный регион:

- Краснодарский край.

Ограничение:

- регион нельзя хардкодить в модулях, фильтрах, аналитике и dashboard;
- тестовый регион должен приходить через registry или adapter;
- будущие регионы должны добавляться без изменения бизнес-логики.

### Sprint 1.2. Auth-ready режим без реальной авторизации

До полноценной авторизации платформа может работать с mock-профилем
пользователя, но интерфейсы и API должны быть готовы к реальному пользователю.

Ограничение:

- не писать новую логику через прямой `role === "admin"`;
- использовать adapter/guard для проверки роли, прав, workspace и региона;
- UI-скрытие кнопок не является проверкой доступа.

### Sprint 2.1. Planned-модули будущей платформы

`PlatformModuleRegistry` должен поддерживать planned-модули, которые еще не
реализованы как полноценные бизнес-модули, но уже видны в архитектуре,
дашборде и правах доступа.

Обязательный planned-модуль:

- `MODULE_04_WORKS_COSTS` — работы, трудозатраты и стоимость работ.

Planned-модуль не должен создавать ложное ощущение готового функционала. В UI
он отображается как запланированный раздел с паспортом, связями и статусом.

## Что не делать

Не хардкодить роли.

Не хардкодить права.

Не хардкодить регион.

Не использовать номер модуля как системный идентификатор в новой логике после
завершения registry/auth-ready спринтов.

Не делать один общий дашборд.

Не связывать права только с интерфейсом.

Не давать доступ к данным через скрытие кнопок.

Все проверки доступа должны выполняться на сервере.
